Cookie和工作階段
cookie標記的配置問題，cookie可能遭到攻擊
可能遭到攻擊的原因: 安全、HttpOnly和域名

Http是一種無狀態協議。因此Cookie通過存儲與用戶操作相關的值來跟蹤用戶的狀態。這些cookie值將發送到服務器或從服務器發送，並存儲在客戶端的瀏覽器中。

如何保護COOKIE，設置COOKIE以限制其使用時，可以添加以下的標誌
SECURE - Avoid transmission over an insecure channel
HTTPOnly - 不允許javascript讀取cookie的值
domain - 設置cookie可用的域
Path - 設置cookie可用的子文件夾和頁面
EXPIRE - 確定何時應刪除COOKIE

用戶端安全措施
不要完全依賴客戶端機算，應該要在SERVER端執行他們
不要在客戶端存儲密碼等敏感信息

XSS攻擊(跨站腳本攻擊): 允許攻擊者在用戶的瀏覽器中執行惡意腳本。攻擊者通常通過注入惡意代碼來竊取用戶信息，操控會話或修改網頁內容
大致分成幾種類型
Stored XSS (儲存型)
Reflected XSS (反射型)
DOM-Based XSS (基於 DOM 的類型)

不信任任何輸入的資訊
操作系統命令注入:將SHELL指另加到請求的參數值從而執行到惡意的程式碼

避免信任無輸入的相關漏洞
1. 永遠不要相信用戶輸入
2. 在人為輸入的時候限制用戶的選項(使用下拉選單或其他可以控制輸入的方式)
3. 在使用前使用安全驗證方案驗證所有輸入，其中包括來自文件，其他服務或數據庫的輸入。

使用以下方式執行服務器端驗證
1. 完全匹配
2. 白名單
3. 黑名單
拒絕無效數據、否則至少要清除或轉義

最低權限

故障時的安全性
確定故障的區域
1. 除非用戶已明確鳩收到應用程序某個部分的權限，否則應拒絕他們訪問
2. 所有操作都應具有確定的結果，並且必須使用通用錯誤消息處理異常

每個程式碼塊應該只有三個確定的結果
1. 用戶獲得授權，則執行操作
2. 用戶未獲得授權則不得操作
3. 發生異常，則回滾操作並顯示錯誤消息
